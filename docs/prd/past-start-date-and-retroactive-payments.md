# PRD: 투자 시작일 소급 입력 및 과거 납입 기록 수동 체크

| 항목 | 내용 |
|------|------|
| **기능명** | 과거 시작일 지원 + 소급 납입 기록 |
| **우선순위** | P1 |
| **한 줄 설명** | 앱 이전부터 투자해온 사용자가 실제 시작일을 등록하고 과거 납입 내역을 소급 입력할 수 있도록 지원 |
| **버전** | 1.0 |
| **작성일** | 2026-04-21 |

---

## 1. 배경 및 문제 정의

### 1.1 현재 상황

사용자가 투자 항목을 등록할 때 `start_date`를 과거 날짜로 설정하면, 앱 내부에서 두 가지 날짜 개념이 혼재된다.

| 날짜 필드 | DB 컬럼 | 역할 |
|---|---|---|
| 실제 투자 시작일 | `start_date` (optional) | 진행률·경과 기간 계산 기준 |
| 앱 항목 생성일 | `created_at` (자동) | 납입 기록 추적 시작점 (현재 혼용 중) |

### 1.2 핵심 버그

`getPaymentHistoryFromStart()`가 `start_date`를 기준으로 납입 기록을 생성한다. 즉, **2024년 6월에 투자를 시작했고 2025년 4월에 앱을 처음 등록한 사용자**는 아래 상황을 겪는다.

```
납입 기록 (현재 잘못된 동작)
─────────────────────────────
2024년 6월  ● 미납 (빨간불) ← 실제로 납입했지만 앱이 모름
2024년 7월  ● 미납 (빨간불)
...
2025년 3월  ● 미납 (빨간불)  ← 10개월 전부 빨간불
2025년 4월  ● 납입 완료 ✓
```

**사용자 심리:** "나는 한 번도 안 빠졌는데 앱이 전부 미납으로 표시한다" → 신뢰 붕괴 → 이탈

### 1.3 해결 방향

두 날짜를 역할에 따라 분리한다.

- **진행률·경과 기간**: `start_date` 기준 (실제 투자 시작일)
- **납입 기록 자동 추적**: `created_at` 기준 (앱 항목 생성일)
- **소급 구간**: `start_date` ~ `created_at` → 사용자가 수동으로 체크 가능

## 2. 사용자 시나리오

### 시나리오 A. 과거 시작일 설정 + 소급 납입 기록 입력

> 2024년 6월 1일부터 삼성전자 ETF에 투자해왔고, 2025년 4월에 이 앱을 처음 사용하는 직장인

1. 투자 항목 추가 시 `투자 시작일`을 **2024년 6월 1일**로 설정
2. 앱이 "2024년 6월 ~ 2025년 3월은 앱 이전 기간입니다. 납입 내역을 기록하시겠어요?" 안내
3. 사용자가 소급 구간을 월별로 탭하여 납입 완료 표시
4. 이후 2025년 4월부터는 기존처럼 자동 추적

### 시나리오 B. 앱 이전 기간 기록 안 함

1. 동일하게 과거 시작일 설정
2. 소급 기록 안내를 스킵
3. 진행률은 실제 시작일 기준으로 정확하게 표시 (8%)
4. 납입 기록은 `created_at` 이후부터만 표시
5. 소급 구간은 "추적되지 않은 기간"으로 회색 표시

## 3. 기능 명세

### 3.1 진행률 / 경과 기간 계산 — 변경 없음

`useInvestmentCalculations.ts`는 그대로 `start_date` (없으면 `created_at`) 기준.

```
진행률: start_date 기준으로 계산 ← 변경 없음
경과: "10개월째 도전 중 🔥" ← start_date 기준 ← 변경 없음
```

### 3.2 납입 기록 생성 기준 — 핵심 변경

`useInvestmentData.ts`에서 `getPaymentHistoryFromStart` 호출 시 기준 날짜 분리.

| 구분 | 기존 | 변경 후 |
|---|---|---|
| 자동 추적 시작점 | `start_date ?? created_at` | `created_at` 고정 |
| 소급 구간 | 없음 | `start_date` ~ `created_at` 별도 생성 |

### 3.3 납입 기록 화면 구성

```
납입 기록
─────────────────────────────────────────
[소급 기록 · 앱 등록 이전 기간]        ← 회색 섹션 헤더

  2024년 6월  ○ 탭하여 기록 가능
  2024년 7월  ✓ 납입 완료 (소급)
  ...
  2025년 3월  ○ 탭하여 기록 가능

[자동 추적 · 2025년 4월부터]           ← 기존 녹색 섹션 헤더

  2025년 4월  ✓ 납입 완료
  2025년 5월  ✓ 납입 완료
```

#### 소급 기록의 시각적 구분

| 항목 | 자동 추적 기록 | 소급 기록 |
|---|---|---|
| 배경색 | 기본 카드 | 회색/muted |
| 체크 아이콘 | 초록 (brand color) | 회색 |
| 월 라벨 | 일반 | "(소급)" 표시 |
| 납입일 세부 날짜 | 기록됨 | 기록 불가 (월 단위만) |

### 3.4 소급 납입 데이터 저장

소급 기록도 동일한 `payment_completions` 테이블(또는 동등한 Supabase 테이블)에 저장하되, `is_retroactive: boolean` 컬럼을 추가하여 구분.

| 컬럼 | 타입 | 설명 |
|---|---|---|
| `investment_id` | uuid | 투자 항목 ID |
| `year` | int | 납입 연도 |
| `month` | int | 납입 월 |
| `day` | int? | 납입일 (소급 시 null) |
| `is_retroactive` | bool | 소급 기록 여부 |

### 3.5 온보딩 안내 (과거 시작일 설정 시)

투자 항목 추가 폼에서 `start_date`가 `created_at`보다 과거로 설정되면 아래 Sheet/Toast를 표시:

```
💡 앱 이전 기간이 있어요

2024년 6월 ~ 2025년 3월 (10개월)은
앱에서 자동 추적되지 않았던 기간이에요.

이 기간의 납입 내역을 직접 기록하시겠어요?

[지금 기록하기]   [나중에]
```

"지금 기록하기" → 투자 상세 화면의 납입 기록 탭으로 이동, 소급 구간 펼침

## 4. 엣지 케이스

| 케이스 | 처리 방법 |
|---|---|
| `start_date`와 `created_at`이 같은 달 | 소급 구간 없음, 기존 동작 그대로 |
| `start_date`가 `created_at`보다 미래 | start_date를 무시하고 created_at 사용 (또는 경고) |
| 소급 기간이 매우 긴 경우 (5년 이상) | 제한 없음, 단 UX 주의 (스크롤 긴 목록) |
| 소급 기록 후 start_date 변경 | 기존 소급 기록은 유지, 새 소급 구간 재계산 |

## 6. 구현 단계

```
1단계: 날짜 분리 (버그 수정, 빠른 배포)
  → useInvestmentData.ts 한 줄 수정
  → created_at 기준으로 자동 추적

2단계: 소급 구간 UI 표시
  → 납입 기록 화면에 소급 구간 회색으로 표시 (체크 불가)
  → "추적되지 않은 기간" 라벨

3단계: 소급 납입 수동 체크 기능
  → DB 마이그레이션 (is_retroactive 컬럼)
  → 소급 구간 탭하여 체크/언체크
  → 저장 API 연동

4단계: 온보딩 안내 Sheet
  → 과거 시작일 설정 시 소급 안내 노출
```
